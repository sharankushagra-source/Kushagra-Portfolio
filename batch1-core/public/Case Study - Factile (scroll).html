Case study · Product design

Customization was powerful. Using it was the hard part.

Preserve what users know. Remove what slows them down. And tell the difference between debt you fix — and debt you expose.

RoleProduct designer — end-to-end UX & UI
SurfaceFactile · the Customize experience
ContextMature product · older & long-term users
FocusDiagnosis · interaction · product model

The screens here are pre-release iterations from my design process — I'm not authorized to show Factile's shipped production build. So this case study is about the reasoning and the iterations behind the work, not a claim on final shipped pixels.

The constraint

A cleaner interface couldn't cost users their familiarity.

Older & long-term users. Relocating familiar controls carries real relearning cost.
Four learned zones stay put — top nav, settings sidebar, scope controls, and the workspace.
So the brief wasn't “redesign it.” It was improve the experience inside a structure people already trust.

Diagnosis · the spine of this case study

The interface carried three different kinds of debt.

Not one broken thing — three layers, each demanding a different response. This framework organizes everything that follows.

01 · UI debt
Visual language
Consistency & readability — the surface layer.
  • Inconsistent icons & radii
  • Weak type hierarchy
  • Flat elevation & contrast
02 · Interaction debt
Everyday friction

  • Overloaded navigation
  • Toast overload
  • Ambiguous fields
03 · Product-model debt
Mental model
How customization itself is structured.
  • Scope decided too early
  • Global vs. game confusion
  • Overrides before intent

02
Interaction debt

Routine actions demanded more attention than they deserved.

Each friction point was small. But customization is a high-frequency, settings-heavy workflow — so small interruptions compound.

Overloaded navigation. 11+ ungrouped sidebar items to scan every visit.
Weak active state. The current section relied mostly on colour alone.
Repeated toasts. Every save & reset fired its own interruption.
Ambiguous fields. Game selection & settings search looked nearly identical.
Unclear scope. Global / Game-Specific state was easy to miss.
Unrestricted colour. A raw picker placed contrast risk on the user.
03
Product-model debt

The hardest problem hit before a setting ever changed.

Before touching anything, the model forces one branching decision — scope — and hides three extra steps behind one of the answers.

User wants to change one setting
Global or Game-Specific?
If Global
Find the setting
Change it
2 steps
If Game-Specific
Choose a game
Enable override
Find the setting
Change it
4 steps
Asked too early. The scope question comes before the user has any intent.
The branch is invisible. Nothing warns that “Game-Specific” hides three extra steps.
The approach

Improve what could safely change. Preserve what users already knew.

Preserve

Overall layout · navigation location · the customization structure people had already learned.

Improve

Hierarchy · grouping · feedback · affordances · accessibility · consistency — everything that could change without moving the furniture.

3 · Turn a hidden dependency into a visible sequence.

Why it confused
Three steps depended on each other — but nothing said so, or in what order.
The game field looked pre-filled, so users couldn't tell if they'd chosen anything.
What got clearer
The three steps became an explicit, numbered sequence.
Each control stays disabled until the previous step makes it relevant.

Honest reflection

Improving the current model didn't make it the best model.

The constraint justified preserving the current model for this pass — it didn't put that model beyond criticism. Grouped by the layer each one lives in:

Product-model
Scope, still too early
Users still choose Global vs. Game-Specific before changing anything — ideally they'd change first, then choose where it applies.
UI
Both panels on white
Sidebar & workspace still share a white background — weak zoning for a first-time older user.
Interaction
No outcome preview
Abstract settings still change without showing their effect immediately.
UI · Accessibility
Unproven colour safety
Curated swatches aren't yet validated as contrast-safe pairs.
Interaction
Undefined failure states
Autosave failure & offline behaviour remain unspecified.
Product-model
The override is a gate, not a result
To customize one game, users must first switch on “game-specific settings” before changing anything. Ideally, editing a setting for that game would create the exception on its own — the override should follow intent, not precede it.
Future direction · hypothesis, not shipped

What if users customized first — and chose scope only when it mattered?

Delay the scope decision until there's something to apply. It should be validated with existing users before it replaces the familiar model.
